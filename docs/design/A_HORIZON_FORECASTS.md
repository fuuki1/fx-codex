# 設計A: マルチホライズン予測トラック（分析時間足と予測ホライズンの分離）

- ステータス: **設計のみ**（実装・マージは合格基準クリア後に別途承認）
- 前提監査: reports/learning-loop-audit-trader-mini-20260716-{72h,24h}.md（E2Eループ稼働実証済み）
- 関連: 次段階設計項目 2/3/4（ホライズン分離・多ホライズン・較正管理）、項目5（shadowゲート）、
  項目6（purged CV/embargo）とはインターフェースのみ定義。項目7（時間軸統合）は本設計の出力を消費する別設計。

## 1. 目的と非目的

**目的**: 5分ごとの各更新で、通貨ペアごとに複数の予測ホライズンの予測を同時生成し、
それぞれに (a) 方向、(b) 較正済みの上昇/下落/横ばい確率、(c) 予想価格の中央値と価格帯、
(d) 予想値幅、(e) 確信度 を出力する。満期後は方向的中に加えて価格誤差・分位点損失・
較正品質（Brier / log loss）・コスト控除後Rを自動採点し、symbol×horizonセル単位で学習する。

**非目的**:
- 既存の時間足別トラック（15m/1h/4h/1d、稼働中の本番ループ）の置換・変更はしない。並走する。
- 発注・執行は恒久的に対象外（分析・意思決定支援専用の境界は不変）。
- 複数ホライズンの「統合判断」は本設計の範囲外（項目7で別途設計）。

## 2. ホライズン集合

構成可能とし、初期セットは直近仕様の8本を採用する:

| label | horizon_hours(市場オープン時間換算) | 採点許容誤差(±h) | 学習間引きgap(h) | ML間引きgap(h) |
|---|---|---|---|---|
| 15m | 0.25 | 0.10 | 0.25 | 1.0 |
| 30m | 0.50 | 0.15 | 0.50 | 2.0 |
| 1h  | 1.0  | 0.25 | 1.0  | 4.0 |
| 3h  | 3.0  | 0.50 | 1.5  | 6.0 |
| 6h  | 6.0  | 1.0  | 3.0  | 12.0 |
| 12h | 12.0 | 2.0  | 6.0  | 24.0 |
| 24h | 24.0 | 2.0  | 12.0 | 48.0 |
| 3d  | 72.0 | 6.0  | 36.0 | 144.0 |

- 「3市場日」= 市場オープン時間換算72h（週末49hは時計停止、既存 `open_hours_between` の規約を踏襲）。
- 5m ホライズンは本設計に含めない。追加する場合は恒久shadow（項目5のゲート設計）を先に通す。
- 間引きgapは「horizon/2、ただし下限=記録周期」を原則にする。5分周期×長ホライズンの
  評価窓重複（24hで約96%）によるサンプル水増しを抑える。gap値は定数表で管理し、
  監査ツールが実効サンプル数（間引き後）を常時報告する。

## 3. アーキテクチャ: 予測の生成

### 3.1 分離の原則

現行は「分析時間足=ホライズン」（15m足→15分後）が固定結合している。本設計では:

- **分析時間足（15m/1h/4h/1d のテクニカル・レーティング等）は特徴量**になる。
- **ホライズンは予測対象**になり、(symbol, horizon) ごとに独立した1予測を生成する。

### 3.2 生成器（fx_intel/horizon_forecast.py 新設・純ロジック）

```
build_horizon_forecasts(symbol, tech, currency_scores, windows, news_items,
                        macro_features, now, profile_lookup, band_provider,
                        horizons=DEFAULT_HORIZONS) -> list[HorizonForecast]
```

- 方向スコア: 既存 `timeframe._tf_direction_score` と同系の複合だが、**時間足レーティングの
  重みをホライズン別に持つ**。初期プライア（学習前の既定）:

| horizon | 15m | 1h | 4h | 1d | news |
|---|---|---|---|---|---|
| 15m/30m | 0.45 | 0.25 | 0.10 | 0.05 | 0.15 |
| 1h/3h   | 0.25 | 0.35 | 0.15 | 0.05 | 0.20 |
| 6h/12h  | 0.10 | 0.25 | 0.30 | 0.10 | 0.25 |
| 24h/3d  | 0.05 | 0.15 | 0.30 | 0.25 | 0.25 |

  学習開始後は既存 `_estimate_weights` と同じシュリンク付き再推定を
  (symbol×horizon) セルで行う（サンプル20件ガード等の既存定数を踏襲）。
- マクロ特徴（金利/VIX/DXY・リスクレジーム）: 6h以上のホライズンの特徴量に接続
  （C群の課題を本設計で同時に解消。TTLキャッシュの鮮度メタデータを特徴量に併記し、
  stale時はニュートラル値+品質減点でfail-soft）。
- ゲートは既存と同一を全ホライズンに適用: 鮮度fail-closed・イベント警戒窓(standby)・
  週末(closed)・データ品質・期待値ガード（セル=symbol×horizon×directionで発動）。

### 3.3 確率出力（p_up / p_down / p_flat）

- 定義: 満期時の符号付き変動 move に対し、|move| ≤ 0.1×ATR_h を flat、それ以外を符号で up/down
  （既存 hit/miss/flat と同じATR比、ATR_h はホライズン適合ATR: 下記3.4）。
- v1（学習前〜少サンプル）: 複合スコアと状態バケットから **ビン較正**で確率化。
  セルの採点済みサンプルが n<50 の間は `calibrated=false` を明示し、確率は
  「クラス基底率へのシュリンク値」を出す（画面には未較正マークを必ず表示）。
- v2: n≥50 で単調較正（isotonic相当の区分定数、依存ゼロ実装）。
- v3（別設計との接続）: 未マージの3ヘッドGBDT（realized_net_r ブランチ）をホライズン別に
  学習させ、確率ヘッドを置換。契約は `horizon-pit-v1`（下記5章）を必須とする。
- 制約: p_up + p_down + p_flat = 1、各∈[0,1]。丸めは出力層のみで行い内部は浮動小数。

### 3.4 価格帯出力（p10 / p50 / p90・予想値幅）

- v1は**状態条件付き経験分位点**（モデルレス・誠実表示）:
  - 過去の同 (symbol, horizon, ボラバケット[既存atr_pct 3分位], セッション) の
    実現 move 分布から p10/p50/p90 を取る。
  - p50 に方向スコアによるシフトは**入れない**（v1は「値幅の予報」を主とし、
    中央値の方向性は方向確率に委ねる。点予測の过剰な見かけ精度を避ける）。
  - 予想価格帯 = close + [p10, p90]、予想値幅 = (p90 − p10)。
  - セル標本 n<40 はボラバケットを外して縮退、n<20 はホライズン全体分布へ縮退
    （縮退段階を `band_source` として行に記録）。
- ATR_h（flat判定・帯の正規化用）: 参照時間足のATRをホライズンへスケール
  （√時間則を初期値とし、実現値幅での再推定を学習で上書き）。
- v2: 分位点ヘッド（3ヘッドGBDT）で置換。置換後も経験分位点はベースラインとして
  併記し、pinball lossで優劣を常時比較（負けているモデルは使わない）。

## 4. 記録スキーマとPIT契約

### 4.1 新ジャーナル（追記専用・既存ファイルには一切触れない）

`logs/briefing_horizon_forecasts.jsonl` — 1行 = 1 (symbol, horizon, cycle)。
3ペア×8ホライズン×5分周期 = 288行/時間 ≈ 3〜4MB/日（許容。ローテーションは運用課題として明記）。

```json
{
  "schema_version": 1, "contract": "horizon-pit-v1",
  "ts": "...", "prediction_time": "...", "source_cutoff": "...",
  "max_feature_available_time": "...", "pit_eligible": true,
  "symbol": "USDJPY", "horizon": "3h", "horizon_hours": 3.0,
  "direction": "long|short|neutral|standby|closed",
  "p_up": 0.41, "p_down": 0.33, "p_flat": 0.26, "calibrated": false,
  "close": 162.163, "atr_h": 0.41,
  "band_p10": -0.35, "band_p50": 0.02, "band_p90": 0.44, "band_source": "vol_bucket",
  "expected_range": 0.79, "conviction": 27,
  "features": {"rating_15m":..., "rating_1h":..., "rating_4h":..., "rating_1d":...,
                "rsi": ..., "adx": ..., "ma_gap_atr": ..., "atr_pct": ...,
                "news_score": ..., "news_count": ..., "session": "tokyo|london|ny|ldn_ny|off",
                "macro_risk": ..., "vix": ..., "dxy_chg": ...},
  "gates": {"freshness_ok": true, "event_window": false, "market_open": true},
  "weights": {"w_15m":..., "w_1h":..., "w_4h":..., "w_1d":..., "w_news":...},
  "generator_version": "hf-1"
}
```

- PIT順序 `source_cutoff ≤ max_feature_available_time ≤ prediction_time` を
  journal.append 時に検証し、違反は `PointInTimeError` で書込み拒否（既存fusion-pit-v1と同型）。
- 中立・standby・ブロックされた行も**必ず記録**する（B群のshadow還流の土台。
  本設計では記録まで、学習への還流方式はB設計で確定）。
- writer所有権: `com.fx-codex.briefing`（既存の5分実行内で生成。ネットワーク取得は
  既存分を共有し追加取得ゼロ、feature_available_time も共有）。

### 4.2 採点

- 将来価格: 既存の密な5分価格系列 `briefing_tf_prices.jsonl` と結合し、
  「市場オープン時間換算でホライズン±許容誤差、最近傍1点」の既存規約で解決
  （learning.evaluate_history と同一アルゴリズム。全ホライズンで同一コードパス）。
- 採点項目（1予測につき全て記録可能なもの）:
  - 方向: hit / miss / flat（ATR_h×0.1閾値）
  - 確率: 実現クラス vs (p_up,p_down,p_flat) → Brier(3クラス) / log loss
  - 価格帯: 実現moveのp10-p90内包含（カバレッジ、目標80%）と pinball loss(p10/p50/p90)
  - 値幅: |実現move| vs expected_range の比
  - パス: MFE / MAE（5分系列から。既存trade_outcome流儀）
  - コスト控除後R: 実測スプレッド（bid/ask収集済み系列）を執行コストとして控除
    （短いホライズンほどスプレッド支配的になる事実を可視化する。これが5m/15mの
    「見かけ的中率は良いが純Rは負」を炙り出す装置になる）
- 採点はステートレス再計算（現行流儀）。ダッシュボード・監査ツールが同一関数を共有。

## 5. 学習（symbol × horizon セル）

- `fx_intel/horizon_learning.py` 新設: tf_learning と同じ「実績あるコアを薄く再利用」方式。
  - スライス: (symbol, horizon) → learning.evaluate_history（ホライズン&許容誤差を注入）
  - 学習: learning.derive_profile 再利用（重み再推定・確信度較正・状態×方向補正・Brier・反省）
  - 追加: 3クラス確率の較正テーブル、経験分位点テーブル、実効サンプル数（間引き後）の常時記録
- サンプル数ガード: 既存定数（20/8/12）を踏襲し、**ホライズン別に到達予測日数を
  ダッシュボードへ表示**（15mは数日、3dは数週間かかることを最初から明示して
  「動いていない」誤解を防ぐ）。
- 多重検定の規律: **各ホライズンは自分の主ラベルのみで学習**（補助ホライズン採点はしない）。
  ホライズン間の比較・統合は項目7の統合設計のみが行う。
- CV/評価（項目6との接続）: walk-forward分割で purge=horizon、embargo=horizon を
  最低要件としてインターフェース定義（実装は項目6）。

## 6. shadow運用と昇格ゲート（項目5の适用）

- 新トラックは**全ホライズンshadowで開始**: Discord通知なし・委員会/既存判断への影響ゼロ・
  ダッシュボードに「shadow」バッジ付きで表示のみ。
- セル単位の昇格条件（表示を「参考」→「本採用」に切替える条件。発注は存在しないので
  昇格の意味は「統合設計(項目7)への入力資格」に限定）:
  1. 間引き後 n ≥ 100（15m〜3h）/ n ≥ 60（6h〜24h）/ n ≥ 40（3d）
  2. 3クラスBrier ≤ 基底率予測のBrier（クライマトロジーに勝つ）
  3. p10-p90カバレッジ ∈ [70%, 90%]
  4. コスト控除後期待R > 0 …を満たさない場合は「予測は表示するが統合には不使用」
  5. データ品質・鮮度違反ゼロの直近7日
- 降格: 直近ローリング窓で 2 または 4 を割ったセルは自動でshadowへ戻す（履歴は保持）。

## 9. リスクと限界（最初から明示する）

1. **長ホライズンのラベル重複**: 間引きとpurged CVで抑えるが、3dの実効サンプルは
   名目の1/400程度になる。3dセルの学習成立は月単位である事実をUIに常時表示する。
2. **v1価格帯はクライマトロジー**（状態条件付き経験分布）であり「モデルの予言」ではない。
   band_source を行に刻み、ダッシュボードでも「経験帯」と表示する。
3. **短ホライズンはスプレッド支配**: 15m/30mはコスト控除後Rが構造的に負になりやすい。
   これは欠陥ではなく事実の可視化であり、昇格条件4が自然にフィルタする。
4. 確率較正はセル単位の少サンプル期間が長い。未較正マークの表示を必須とし、
   較正前の確率を統合(項目7)に使うことを禁止する。
5. 生成行数は約9倍になる。ログローテーションと監査の実効n表示を運用チェックリストに追加。
